**Docker コンテナが起動しない**ときは、「ビルド失敗」なのか「起動直後のクラッシュ」なのかを先に切り分けると原因が特定しやすいです。今回のケースは後者でした。

## 1. まず全体像を確認する

```bash
docker compose ps -a
```

確認ポイント:

- **Status** が `Exited (1)` など → 起動はしたがすぐ落ちている
- **Created だけで Up にならない** → ビルド失敗や起動コマンドエラーの可能性
- **Up になっているのにアクセスできない** → ポート設定やアプリ側の問題

## 2. ログを見る（最重要）

```bash
docker compose logs backend
```

直近だけなら:

```bash
docker compose logs backend --tail 50
```

リアルタイムで追うなら:

```bash
docker compose up backend
```

今回のように **Python の Traceback** や `NameError` が出ていれば、原因はほぼアプリコード側です。Dockerfile / compose ではなく **`main.py` の import 漏れ** が直接の原因でした。

## 3. ビルドと起動を分けて確認する

### ビルドだけ

```bash
docker compose build backend
```

- ここで失敗 → **Dockerfile**、**COPY パス**、**requirements.txt** を疑う
- 成功する → イメージ作成は問題なし

### 起動だけ

```bash
docker compose up backend
```

- ビルド成功 + 起動直後に落ちる → **アプリコード**、**CMD**、**環境変数** を疑う

## 4. コンテナ内の状態を直接確認する

コンテナがすぐ落ちる場合は、シェルで中に入って確認します。

```bash
docker compose run --rm backend bash
```

中に入ったら:

```bash
ls -la /app
python -c "import main"
uvicorn main:app --host 0.0.0.0 --port 8000
```

今回なら `python -c "import main"` の時点で `NameError: name 'CORSMiddleware' is not defined` が再現できたはずです。

## 5. 設定ファイルを疑う順番

切り分けの目安:

| 症状 | まず見る場所 |
|------|-------------|
| `docker compose build` で失敗 | `Dockerfile`, `COPY` パス, `requirements.txt` |
| ビルド成功、起動直後に落ちる | アプリコード, `CMD`, ボリュームマウント先 |
| 起動はするが接続できない | `ports`, ホスト側ポート競合, バインドアドレス |
| コード変更が反映されない | `volumes` のマウント先 |

今回の構成では:

- **Dockerfile**: 依存関係インストール → OK
- **docker-compose.yml**: `./src/backend:/app` でコードマウント → OK
- **main.py**: `CORSMiddleware` 未 import → NG

## 6. よくある落とし穴

- **ログを見ずに Dockerfile だけ疑う**  
  今回は Dockerfile は正常で、ログを見ればすぐ特定できました。
- **`docker compose up -d` だけ使う**  
  バックグラウンド起動だとエラーが見えにくいです。最初は foreground で起動するのがおすすめです。
- **ホストで動く = コンテナでも動く、と思い込む**  
  ボリュームマウントしている場合、ホストの `main.py` がそのまま使われます。コンテナ内で import エラーが出るなら、ホスト側のコードを直せば解決します。

## 7. 実践的な調査手順（今回用）

1. `docker compose up backend` を foreground で実行
2. Traceback を読む
3. エラー行（`/app/main.py`, line 12`）を特定
4. 該当ファイルを開いて import や変数定義を確認
5. 修正後、再起動

```bash
docker compose up backend
```

`--reload` 有効なら、ファイル修正後は自動で再読み込みされます。

---

**最短ルート**はこれです。

```bash
docker compose up backend
```

ログの **最後の Traceback 1行** を読む → そのファイル・行番号を開く。

Docker 周りの問題なら `build failed` や `COPY failed` が出ます。アプリの問題なら `NameError`, `ModuleNotFoundError`, `ImportError` など Python のエラーが出ます。今回は後者でした。

必要なら、`frontend` 側が API に繋がらないときの調査手順も整理できます。
